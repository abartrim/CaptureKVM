Task: Add KVM-over-IP support to CaptureKVM using VS Code/Copilot, Go-only implementation

Repository:
https://github.com/abartrim/CaptureKVM

Goal:
Extend CaptureKVM with optional KVM-over-IP support while preserving the existing local-first architecture, but use a binary UDP streaming protocol for the hot path because latency matters more than strict reliability.

Current/local model:
- macOS CaptureKVM app uses a local UVC capture dongle for video.
- The macOS app renders the freshest frame directly with `AVSampleBufferDisplayLayer` and drops late frames.
- The macOS app forwards keyboard and mouse state to an ESP32-S3 over USB serial or BLE using tiny binary frames.
- The ESP32-S3 presents as USB HID keyboard and mouse to the target computer.
- The current local input path is already latency-oriented:
  - keyboard sends an 8-byte USB boot keyboard report on every state change
  - mouse sends a 4-byte USB boot mouse report
  - mouse deltas are coalesced on a 4 ms timer, about 250 Hz
  - serial runs at 921600 baud
  - serial and BLE both carry COBS-framed packets with CRC8

New/remote model:
- Raspberry Pi or Linux host uses a UVC HDMI capture dongle for video.
- Raspberry Pi or Linux host runs a new Go-based `capturekvm-agent`.
- `capturekvm-agent` exposes an authenticated control plane over HTTP, but uses binary UDP for low-latency video and input streaming.
- Remote clients primarily means the existing macOS app.
- A browser UI may exist for diagnostics or fallback testing, but it is not the primary low-latency path because browsers cannot participate in the same raw UDP design.
- Input is forwarded to:
  1. the existing ESP32-S3 over USB serial, or
  2. an experimental Raspberry Pi USB gadget HID backend.

Important implementation constraints:
- Make the new server or agent implementation in Go only.
- Do not add Python, Node, Rust, Java, or shell-heavy server code.
- Shell scripts are acceptable only for install or setup helpers and must not contain core server logic.
- The macOS app remains Swift and macOS as it is today. The new KVM-over-IP agent is Go.
- Primary deployment target is Ubuntu on Raspberry Pi 4 and newer, including Raspberry Pi 5.
- macOS is an acceptable development and bring-up host for the Go agent, but not the primary deployment target.
- Prefer boring, explicit code over large protocol frameworks.
- Latency is the primary optimization target.
- Do not add reliable-by-default transport layers on the hot path.

Reference decisions:
- Do not build a full PiKVM clone.
- Do not fork TinyPilot or PiKVM.
- Do not use WebSocket or JSON on the video or live input hot path.
- Keep HTTP only for setup, auth, session negotiation, health, and status.
- Use binary UDP datagrams for remote input.
- Use binary UDP video streaming, ideally H.264 over UDP with minimal buffering.
- Protect UDP payloads with authenticated encryption.
- Use AES-256-GCM for v1 so the Swift client can use CryptoKit and the Go agent can use the standard library.
- Treat AES-GCM as a compatibility choice across macOS and Ubuntu on Raspberry Pi, not as a guaranteed hardware-accelerated choice on every Pi deployment.
- On macOS Apple silicon, AES-GCM should be hardware-backed.
- On Ubuntu arm64 Raspberry Pi, detect and log CPU AES and PMULL capability at startup instead of assuming acceleration.
- Reuse the existing CaptureKVM HID report model instead of inventing a new semantic input schema.
- Treat the ESP32 serial protocol as the compatibility anchor for the remote agent.

Why:
- The current app and firmware already optimize for freshest-state-wins behavior rather than reliable ordered delivery.
- JSON and WebSocket would add encoding, framing, head-of-line blocking, and backpressure behaviors that work against the current low-latency design.
- The app already uses a rendering path that benefits from compressed video delivered with minimal buffering.
- The firmware already accepts compact fixed-size HID reports, so the remote protocol should stay equally compact.

Desired high-level architecture:

Local mode:
  macOS app
    -> local UVC video source
    -> local ESP32-S3 serial or BLE HID bridge
    -> target computer USB HID

Remote mode:
  target HDMI
    -> UVC capture dongle
    -> Raspberry Pi or Linux host
    -> low-latency encoder or packetizer
    -> capturekvm-agent
    -> UDP video stream to macOS CaptureKVM app

  remote keyboard and mouse
    -> UDP input datagrams from macOS CaptureKVM app
    -> capturekvm-agent
    -> HID backend
       -> ESP32-S3 serial
       OR
       -> Raspberry Pi /dev/hidg* USB gadget endpoint
    -> target computer USB HID

  control and status
    -> HTTPS or HTTP on trusted LAN or VPN
    -> session setup, auth, health, stats, backend state

4. UDP binary protocol

Implement a compact binary protocol.

General requirements:
- One UDP datagram is one protocol packet.
- No JSON on the hot path.
- Include version, session ID, packet kind, sequence number, monotonic timestamp, and payload length in every packet.
- Keep headers fixed-size and simple to parse.
- Keep a small clear header and encrypt only the payload.
- Authenticate the clear header as additional authenticated data.
- Use network byte order.
- Keep packet size below a configurable MTU target, default around 1200 bytes.
- Never rely on retransmission for interactive input.
- Prefer drop-late behavior over buffering.
- Use AES-256-GCM for v1 packet protection.
- Use CryptoKit `AES.GCM` on the macOS side and Go standard library AES-GCM on the agent side.
- Derive or construct a unique 96-bit nonce per packet from session-specific data plus direction and sequence number.
- Reject packets with invalid authentication tags before any payload processing.

Suggested packet header:
- magic: 4 bytes
- version: 1 byte
- packet_kind: 1 byte
- flags: 2 bytes
- session_id: 8 bytes
- seq: 4 bytes
- timestamp_us: 8 bytes
- payload_len: 2 bytes

Encryption requirements:
- Packet form should be `clear_header || ciphertext || gcm_tag`.
- The clear header must be authenticated as AEAD additional data.
- Do not use CRC as a security mechanism on UDP packets.
- The session key is delivered only over the authenticated HTTP control plane.
- Keep v1 to a single crypto suite so the protocol stays debuggable.

Packet kinds:
- input_keyboard
- input_mouse
- input_ping
- video_config
- video_frame
- video_ping
- telemetry

5. Input protocol requirements

Do not model remote input as abstract key down or key up JSON events.
Model it as the same low-level HID reports the local app already emits.

Specifically:
- Keyboard packets should carry the full 8-byte USB boot keyboard report.
- Mouse packets should carry the full 4-byte USB boot mouse report.
- Keep paste as a control-plane action or synthesize it client-side into repeated keyboard reports.
- Reuse the existing semantics:
  - frame type `0x01` means keyboard boot report, payload length 8
  - frame type `0x02` means mouse boot report, payload length 4
  - frame type `0x80` means ping
- The ESP32 serial and BLE transport currently use COBS plus CRC8 because they are stream transports.
- The UDP transport should not wrap packets in COBS.
- The Go agent should translate UDP input packets into the existing serial frame format when using the ESP32 backend.

Sender behavior:
- On every keyboard state change, send the full current 8-byte report immediately.
- While any key is held, periodically resend the current keyboard state at a low backup rate such as 20 to 30 Hz so a dropped packet does not leave keys stuck indefinitely.
- Coalesce mouse movement like the current app does, approximately every 4 ms.
- Include current button state in every mouse packet so button state self-heals after loss.
- Clamp relative mouse deltas to the existing int8 range because that matches the local firmware path.
- Do not ack every input packet.

Receiver behavior:
- Apply only packets for the active session.
- Drop packets older than the newest sequence seen for that packet kind when possible.
- Maintain simple anti-replay checks inside the current session window.
- Expose counters for drops, late packets, malformed packets, and active input rate.
- Fail closed on AEAD authentication failure.

6. Video protocol requirements

The primary remote video path must also be binary over UDP.

Preferred MVP approach:
- Encode H.264 on the Raspberry Pi or Linux host.
- Use low-latency encoder settings:
  - no B-frames
  - short GOP or frequent keyframes
  - limited buffering
  - constant or capped bitrate
- Packetize encoded bytestream chunks into UDP datagrams with sequence numbers and frame identifiers.
- Allow frame fragmentation across datagrams.
- Reassemble only a very small window on the client.
- Drop incomplete or late frames instead of waiting too long.

Swift client requirements:
- Decode H.264 using VideoToolbox or AVFoundation-compatible low-latency APIs.
- Feed decoded frames into the same `AVSampleBufferDisplayLayer` based rendering approach the local path already uses.
- Preserve the current freshest-frame-wins behavior.
- Avoid building a deep decode queue.

Go agent requirements:
- Wrap or manage a practical encoder pipeline, but keep CaptureKVM control logic in Go.
- If an external encoder process is required, it must be treated as an external runtime dependency and supervised by the Go agent.
- Do not regress into HTTP MJPEG for the primary macOS path.

If a fallback HTTP or browser video preview is kept for diagnostics, clearly mark it as a secondary, higher-latency path.

7. HID backend interface

Define a Go interface similar to:

type Backend interface {
    Name() string
    Open(ctx context.Context) error
    Close() error
    Status(ctx context.Context) Status
    SendKeyboardReport(ctx context.Context, report [8]byte) error
    SendMouseReport(ctx context.Context, report [4]byte) error
    Ping(ctx context.Context) error
}

Implement these backends:

A. mock
- Logs reports and counters.
- Used for development and tests.
- Must be the default when running in dev mode.

B. esp32-serial
- Opens a serial port.
- Reuses the existing CaptureKVM ESP32 serial protocol exactly.
- Document the protocol in `docs/protocol.md`.
- Support auto-reconnect.
- Return clear status when disconnected.
- Do not crash the agent on serial disconnect.
- Match the current framing behavior:
  - raw packet is `[type, payload..., crc8]`
  - then COBS encode it
  - then append `0x00`
- Support ping using the existing `0x80` frame and `0xAA` pong behavior.

C. pi-gadget experimental
- Writes HID reports to `/dev/hidg0`, `/dev/hidg1`, or configured endpoints.
- Keyboard and mouse report construction should be isolated and tested.
- If endpoints are missing, backend should fail gracefully with actionable error messages.
- Do not try to fully configure USB gadget mode in the Go service.
- Add docs explaining that Pi gadget setup is separate.

15. Notes for implementation

- Keep remote mode optional.
- Do not assume internet exposure.
- Do local development on macOS first where practical.
- The Go agent should run on macOS for early development so the already-connected capture card and ESP32 can exercise the control plane, serial backend, and initial UDP protocol before Raspberry Pi integration.
- Treat Raspberry Pi as the deployment target, not as a prerequisite for starting implementation.
- Early local macOS testing can use the capture card and ESP32 backend on the same machine, with loopback or LAN UDP endpoints for protocol validation.
- Prefer wired Ethernet on Raspberry Pi.
- Favor packet loss tolerance over retransmission.
- Avoid buffering unless it measurably improves end-to-end latency.
- Use clear logs because this will be debugged with real capture dongles and serial devices.
- Use graceful shutdown.
- Make reconnect behavior robust.
- Keep the code boring and maintainable.
- Reuse the current local report model wherever possible instead of adding a second input model.
